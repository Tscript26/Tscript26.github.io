如何正确实现JAVA单例的热加载

Posted Feb 12, 2026
By Ethan
4 min read
如何正确实现JAVA单例的热加载
如何正确实现JAVA单例的热加载

昨日说到，由于apollo配置错误导致了redis加载日常，并且由于单例导致无法通过apollo实现热重载，那么我应该如何正确的热重载逻辑呢？ 基于这个思考，我开始考虑之前gemini提到的Spring Cloud中的refreshScope。

我参考网上的写法，实现了一下@refreshScope并添加了apollo的监视器用于调用refreshScope，但是运行的时候，骤然发现了大量的加载异常，其中最刺眼的就是Servlet异常。

我明明用的是Webflux，怎么会有Servlet相关的异常？想想我自己的更改：为了实现Bean的热重载，给pom增加了spring-cloud，给方法增加了refreshScope....，会不会是Spring-Cloud带来的？

调查Spring Cloud

针对于Spring Cloud 的背后故事进行了调查，发现大量关于servlet的引用，如果强行为了使用refreshScope而导致干扰，反而得不偿失。

还有什么其他方法？

说到底，我的方案都是针对于单例本身的，但是单例的引用呢？单例的引用是否可以改？

查询发现，通过AtomicReference可以实现属性的热更新

AtomicReference

在 Java 并发编程中，AtomicReference 是 java.util.concurrent.atomic 包下的一个工具类，它提供了一种以原子方式（Atomic）更新对象引用的机制。

简单来说，它是对象引用的“安全外壳”，确保在多线程环境下，对这个引用的“读取”和“赋值”操作是线程安全的，且不需要使用重量级的 synchronized 锁。

AtomicReference 是承载 动态单例（Dynamic Singleton） 的核心组件。它让你能在不停止服务、不阻塞线程的情况下，安全地把旧的 Redis 实例替换为新的。

我使用的是project Reactor的ReactiveRedisConnectionFactory，所以，我只需要重写一个新的用atomicreference包裹的lettucConnectionFactory就可以了。

开始测试

AtomicReference相比较之前的RefreshScope的方式，需要额外实现对ReactiveRedisConnectionFactory的重构，但整体还是非常平滑的。

测试实践，顺利通过。这种方式非常平滑的实现了单例不变，但属性发生变更，实现了redis的切换。

总结

相比较热更新，确保上线前好好进行检查才是更重要的，遵循良好的开发规范，每一条标准规定的背后都不知道是多少个血泪史。另外就是开发前，针对一个问题本身需要进行更细节的调查。想用一个第三方工具，我们不仅需要知道这个工具能做到什么，如果非黑盒的情况下，更是要进行分析，为什么可以，他是怎么做到的。我们虽然不需要重新实现一次，但学习永远是一件必要的事情。
